6. Production bugs found and fixed
==================================

ReviewService:
- Review create previously recalculated taste directly and then InteractionService recalculated again after ReviewCreated interaction creation. Create now relies on the interaction path so taste includes the review interaction and recalculates once.
- Duplicate review upsert previously updated content without taste recalculation. It now recalculates taste once and does not create a duplicate ReviewCreated interaction.
- Review update with UpdatedRatingValue previously recalculated through RatingService and again in ReviewService. It now recalculates once through RatingService.
- Review delete with deleteRating = true previously recalculated once for review deletion and again through rating deletion. It now relies on RatingService for the combined recalculation.
- Review create/update/delete now use transactions around write plus side-effect service calls.
- Deleted-target review deletion is explicit: keeping the rating removes the review and recalculates whole-user taste; deleteRating = true rolls back safely if the deleted target prevents rating deletion.

UserMediaStatusService:
- Status-change interaction recording previously lived in MediaController. UserMediaStatusService now owns the business side effect.
- MediaController no longer duplicates status-change orchestration for POST /api/media/{id}/status.
- Status write plus interaction/taste side effects now run in a transaction.
- Missing/deleted/invalid status attempts create no misleading status interaction.

InteractionService:
- Season interactions now reject seasons whose parent TV media row is deleted.
- Episode interactions now reject deleted episodes, deleted seasons, and deleted parent TV media rows before writing interactions or recalculating taste.

RatingService:
- Season ratings now reject seasons whose parent TV media row is deleted before writing ratings, interactions, aggregates, or taste.
- Episode ratings now reject deleted episodes, deleted seasons, and deleted parent TV media rows before writing ratings, interactions, aggregates, or taste.

7. Side-effect ownership decision
=================================

Review side effects:
- ReviewService owns review writes and decides when a ReviewCreated interaction should be recorded.
- Review create records ReviewCreated through InteractionService; InteractionService owns the post-interaction taste recalculation so review-created interaction points are included.
- Review update/delete do not create ReviewCreated interactions.
- Review update/delete trigger taste recalculation directly through ReviewService when only review presence/content changes.
- Review delete with deleteRating = true delegates the final recalculation to RatingService so rating and review removal are reflected together.

Media status side effects:
- UserMediaStatusService owns MediaStatusChanged interaction recording for user status changes.
- MediaController now calls one service method for user status writes.
- Status changes do not call UserTasteService directly.
- Status changes affect taste indirectly through the MediaStatusChanged interaction recorded by InteractionService.
