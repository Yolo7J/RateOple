2. Architecture decision made and documented
============================================

Decision:
- RateOple is intentionally documented as a pragmatic EF-backed layered architecture, not strict clean architecture.
- RateOple.Core remains an application/service layer over EF Core for now.
- RateOple.Infrastructure continues to own entities, DbContext, configurations, migrations, and seeding.
- The API project owns controllers, auth, middleware, SignalR, HTTP error shaping, and endpoint wiring.
- No broad repository-pattern rewrite or entity migration into Core was introduced.

Architecture rules added:
- Controllers should be thin.
- Services own business rules.
- Infrastructure owns EF Core persistence.
- Core services may use ApplicationDbContext for now.
- Do not introduce repositories unless they solve a real testing/composition problem.
- New business logic needs targeted backend tests.

3. New shared helpers added
===========================

Added:
- ClaimsPrincipalExtensions.GetUserIdOrNull()
- ClaimsPrincipalExtensions.GetRequiredUserId()
- ClaimsPrincipalExtensions.GetUsernameOrNull()

Behavior:
- Optional user ID parsing returns null for missing/invalid NameIdentifier claims.
- Required user ID parsing throws UnauthorizedAccessException with a clear message.
- Controllers no longer keep duplicated local Guid.Parse helpers.

AuthController still reads external/Google claims directly where provider-specific claim handling is required.

4. Exception and ProblemDetails behavior
========================================

Before:
- Controllers frequently caught KeyNotFoundException, ArgumentException, UnauthorizedAccessException, and InvalidOperationException locally.
- Error responses mixed raw strings, anonymous objects, empty NotFound responses, Identity arrays, and ad hoc BadRequest/Conflict responses.
- Several service exceptions were wrapped in generic Exception, losing clean error semantics.

After:
- GlobalExceptionHandler maps common exceptions into application/problem+json.
- Validation errors use ValidationProblemDetails through the ApiController invalid model state factory.
- ProblemDetails includes type, title, status, detail, traceId, and a message compatibility extension.
- Production 500 responses use a generic message; Development may include the exception message.
- Common mappings:
  - ValidationException / ArgumentException -> 400
  - UnauthorizedAccessException -> 403
  - KeyNotFoundException -> 404
  - InvalidOperationException -> 409 when conflict-like, otherwise 400
  - unhandled exception -> 500

Controllers were thinned in the media, ratings, reviews, collections, moderation, notification, profile, follow, discovery, and group user-id paths. Some group controller action-level catches remain and can be removed in a later cleanup pass.

5. DTO validation added
=======================

Validation strategy:
- DataAnnotations, plus custom NotEmptyGuidAttribute and IValidatableObject where cross-field/list validation is needed.
- This matches the existing ASP.NET Core ApiController model validation path and avoids new validation infrastructure.

Covered areas:
- Auth login/register.
- User profile, password change, delete account.
- Media create/update/query DTOs.
- TV season/episode create/update DTOs.
- Tag upsert count.
- Rating/review DTOs.
- Collection create/update/add/reorder/query DTOs.
- Group create/query/post/comment/vote/ban/staff-message/pinned-media DTOs.
- Moderation report/status/query/assignment/audit query DTOs.
- Notification and media status query/update DTOs.

Invalid DTOs now return validation ProblemDetails with errors, traceId, and message.

6. Pagination and query normalization added
===========================================

Added:
- RateOple.Core.Common.Pagination.Normalize()

Rules:
- Page defaults to 1 when lower than 1.
- PageSize defaults to the feature default when lower than 1.
- PageSize is capped at 100.

7. Logging and error cleanup performed
======================================

Changed:
- MediaService now uses ILogger<MediaService> instead of Console.WriteLine.

- Bulk media creation logs per-item failures as warnings while preserving its existing partial-success response contract.
- TestUsersSeeder now throws InvalidOperationException instead of generic Exception when Identity user creation fails.

Remaining intentional catches:

10. Frontend compatibility concerns
===================================

Routes were not renamed.

Potential behavior changes:
- Invalid payloads now fail earlier with validation ProblemDetails.
- Some previously raw-string BadRequest/NotFound/Conflict responses now return ProblemDetails.
- Frontend code that only reads err.response.data.message should still have a compatibility message field on ProblemDetails.
- Frontend code that depends on raw string response bodies should be updated to handle ProblemDetails.
- URL validation on optional image fields may reject malformed non-URL strings that previously reached services.
- Page/pageSize values outside the accepted bounds now receive validation errors at the API boundary.

11. Known limitations and recommended next task
===============================================

Known limitations:
- This is still a pragmatic EF-backed service layer. Core still references Infrastructure by design.
- Some group controller methods still have local exception mapping and should be thinned in a follow-up.
- DTO validation is broad but not exhaustive. More cross-field business validation should stay in services when DB checks are required.
- ProblemDetails tests cover validation shape; explicit 404/500 global handler tests would need either carefully chosen seeded API paths or a test-only endpoint.
- CollectionService still contains PostgreSQL-specific raw SQL for conflict handling, so its service tests need a separate strategy before moving fully to SQLite tests.
- The frontend still has hardcoded API base URLs and raw GUID workflows; those were non-goals for this backend pass.

Recommended next task:
- Continue backend test expansion around GroupService, CollectionService, ModerationService, UserProfileService, and refresh-token rotation using reusable test setup.
